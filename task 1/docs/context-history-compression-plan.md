# План внедрения компрессии истории для агента

## Статус

- Это только план.
- Изменения в прикладной код, схемы API и БД не применялись.
- Реализацию начинать только после вашего отдельного согласования.

## Цель

Реализовать серверный механизм управления контекстом, чтобы агент:

- хранил последние `N` сообщений "как есть";
- заменял более старую часть истории отдельным summary;
- обновлял summary пачками, например каждые `10` сообщений;
- подставлял в upstream-запрос не полную историю, а `summary + последние N сообщений + текущий prompt`;
- снижал расход токенов без потери критичного контекста диалога.

## Текущее состояние

- Сервер хранит историю только как полный список `user`/`assistant` сообщений в SQLite.
- `chatAgent` перед каждым запросом загружает всю историю и целиком отправляет её в модель.
- Клиент знает только про полную историю и локально считает токены по ней.
- `tokenUsage` сейчас не отличает сырой хвост истории от summary.

## Уточнённая постановка

Актор: серверный chat agent.

Триггер: новый пользовательский запрос в существующий диалог.

Ожидаемое поведение:

- если история короткая, агент работает как сейчас;
- если история выросла, агент держит отдельное summary старых сообщений и неизменённый хвост последних `N` сообщений;
- summary обновляется не на каждом запросе, а по порогу, например каждые `10` новых сообщений;
- в модель уходит только эффективный контекст, а не вся когда-либо накопленная история.

## Рекомендуемый подход

### 1. Хранение

Добавить отдельное server-side хранилище summary, не смешивая его с обычными сообщениями.

Рекомендованная форма:

- новая таблица `conversation_summaries`;
- одна актуальная запись на `conversationId`;
- поля:
  - `conversation_id`
  - `summary`
  - `covered_message_id`
  - `covered_message_count`
  - `updated_at`

Смысл:

- `messages` остаётся журналом сырых сообщений;
- `conversation_summaries` хранит отдельный сжатый слой памяти;
- `covered_message_id` фиксирует, до какого сообщения summary уже покрывает историю.

### 2. Эффективный контекст

Перед вызовом модели собирать не полную историю, а структуру вида:

- `summaryMessage`
- `tailMessages`
- `currentPrompt`

Рекомендация по роли summary:

- подставлять summary как синтетическое `assistant`-сообщение с явным префиксом вроде `Conversation summary of earlier turns:`;
- не хранить это summary как обычную запись в таблице `messages`.

Причина:

- summary остаётся частью истории, а не превращается в инструкцию высокого приоритета;
- меньше риск, что summary начнёт конфликтовать с `systemPrompt`.

### 3. Политика компрессии

Серверная политика первой версии:

- `TAIL_MESSAGES_COUNT = N` хранится в конфиге/константе сервера;
- `SUMMARY_BATCH_SIZE = 10` хранится в конфиге/константе сервера;
- если старых сообщений за пределами хвоста меньше порога, summary не пересчитывается;
- после успешного сохранения нового ответа ассистента агент проверяет, накопилось ли достаточно несжатых сообщений для следующего пересчёта.

Рекомендация для первой реализации:

- `10 сообщений` трактовать буквально как `10` записей в истории, а не `10` пар turn'ов;
- `N` не выносить в публичный API, чтобы не усложнять контракт.

### 4. Обновление summary

Summary обновляется только после успешного assistant-turn.

Порядок:

1. сохранить новый `user` message;
2. отправить эффективный контекст в модель;
3. сохранить новый `assistant` message;
4. вычислить, есть ли за пределами `tail` достаточно новых сообщений для компрессии;
5. если да, построить новый summary на основе:
   - предыдущего summary;
   - нового компрессируемого сегмента истории;
6. сохранить обновлённый summary и новый `covered_message_id`.

Такой порядок уменьшает риск рассинхронизации между сырыми сообщениями и summary.

## Необходимые изменения по слоям

### `lib/conversation-store.ts`

Новые обязанности:

- читать summary по диалогу;
- читать только сообщения после `covered_message_id`;
- возвращать эффективный контекст для модели;
- сохранять и обновлять summary отдельно;
- считать количество несжатых сообщений за пределами хвоста.

Новые методы:

- `getConversationSummary(conversationId)`
- `getMessagesAfterSummary(conversationId)`
- `getCompressedContext(conversationId, tailSize)`
- `saveConversationSummary(...)`
- `countCompressibleMessages(conversationId, coveredMessageId, tailSize)`

### `lib/chat-agent.ts`

Изменения в оркестрации:

- заменить загрузку полной истории на загрузку компактного контекста;
- обновить `prepareRequest()` и `buildMessages()`, чтобы они принимали `summary + tail`;
- после успешного assistant-turn вызывать шаг обновления summary;
- одинаково применить это в `respond()` и `stream()`.

### `lib/types.ts`

Добавить отдельные типы:

- `ConversationSummary`
- `CompressedConversationContext`
- `ContextCompressionMeta`

Расширить серверные ответы метаданными компрессии, не меняя входной payload первой версии.

### `lib/token-usage.ts`

Уточнить телеметрию:

- считать токены не по полной истории, а по реально отправляемому контексту;
- отдельно показывать:
  - `historySummary`
  - `historyTail`
  - `historyTotal`

Минимум для первой версии:

- оставить текущую структуру `history`, но добавить `contextCompression` рядом;
- в ideal-варианте расширить разбивку `tokenUsage.request.history`.

### `app/api/chat/*` и `app/api/conversations/[conversationId]/route.ts`

Сами маршруты остаются тонкими.

Но потребуется расширить ответы:

- `POST /api/chat`
- SSE `meta` и `usage`
- `GET /api/conversations/:conversationId`

Полезные поля:

- `contextCompression.enabled`
- `contextCompression.summaryPresent`
- `contextCompression.retainedMessages`
- `contextCompression.compressedMessages`
- `contextCompression.coveredMessageCount`

### `components/chat-console.tsx`

Клиентский риск уже известен: сейчас UI считает токены по полной истории.

Нужно будет:

- отделить `saved history` от `effective context`;
- обновить live estimate, чтобы он не притворялся точным при включённой компрессии;
- отрисовать серверные метаданные компрессии;
- скорректировать подписи вроде `Context: X saved messages`.

### `README.md`

Нужно обновить:

- примеры `tokenUsage`;
- объяснение, что в модель уходит не вся история, а summary + хвост.

## Критерии приёмки

### Функциональные

- При короткой истории агент ведёт себя как раньше и не создаёт summary преждевременно.
- При длинной истории сервер хранит последние `N` сообщений без изменений.
- Старые сообщения заменяются отдельным summary, которое не лежит в обычной истории сообщений.
- В upstream-запрос передаётся summary и только хвост последних `N` сообщений.
- Summary обновляется пачками, а не на каждом запросе.
- `respond()` и `stream()` используют одинаковую логику компрессии.

### Контрактные

- Клиенту доступны метаданные, объясняющие, сколько истории реально ушло в модель.
- `GET /api/conversations/:conversationId` не вводит в заблуждение относительно "сохранённой истории" и "эффективного контекста".
- `tokenUsage` отражает реально отправленный контекст, а не полный журнал переписки.

### Нефункциональные

- Логика компрессии обратима на уровне схемы и не ломает старые разговоры.
- Ошибка обновления summary не должна удалять сырые сообщения.
- При отсутствии summary система корректно работает в fallback-режиме.

## Открытые вопросы

Нужно согласовать до реализации:

1. Какое точное значение `N` нужно для первой версии.
2. Означает ли "каждые 10 сообщений" именно `10` message records или `10` user/assistant пар.
3. Нужно ли клиенту получать полный журнал всех сообщений, или достаточно хвоста + метаданных summary.
4. Должен ли summary строиться локальной функцией, отдельным summarization prompt через ту же модель, или отдельной cheap-моделью.
5. Нужен ли feature flag, чтобы можно было быстро отключить компрессию.

## Границы первой поставки

Включить:

- серверное хранение summary отдельно;
- подстановку `summary + tail` в запрос к модели;
- обновление summary по порогу;
- базовые метаданные компрессии в ответах;
- корректную server-side token telemetry.

Не включать в первую итерацию:

- пользовательскую настройку `N` из UI;
- сложные многоуровневые summary checkpoints;
- пересборку полной истории из summary для клиента;
- автооптимизацию порога по модели.

## Разбиение по агентам

### Агент 1. Backend storage

Зона ответственности:

- `lib/conversation-store.ts`
- миграция SQLite-схемы для summary

Задачи:

- добавить таблицу summary;
- реализовать чтение и сохранение summary;
- реализовать выборку tail и подсчёт compressible messages.

Риск:

- некорректная граница `covered_message_id`.

### Агент 2. Agent orchestration

Зона ответственности:

- `lib/chat-agent.ts`
- `lib/types.ts`
- `lib/token-usage.ts`

Задачи:

- встроить слой compressed context;
- изменить сборку upstream `messages`;
- синхронизировать JSON и SSE режимы;
- вернуть метаданные компрессии и обновлённый token usage.

Риск:

- рассинхрон между записью assistant message и обновлением summary.

### Агент 3. Client and contract

Зона ответственности:

- `components/chat-console.tsx`
- `app/api/conversations/[conversationId]/route.ts`
- `README.md`

Задачи:

- отобразить effective context;
- убрать ложную точность live estimate;
- документировать новый контракт и semantics токенов.

Риск:

- пользователь видит одно количество сообщений, а модель использует другое, без понятного объяснения.

### Агент 4. QA and review

Зона ответственности:

- сценарная проверка и регрессии

Задачи:

- проверить поведение на короткой истории;
- проверить переход через порог summary;
- проверить JSON и SSE;
- проверить восстановление старого диалога;
- проверить ошибки при пустом или повреждённом summary.

1. Согласовать `N`, трактовку `10 сообщений` и формат summary.
2. Внедрить слой хранения summary в БД.
3. Встроить compressed context в `chatAgent`.
4. Обновить token telemetry и серверные метаданные.
5. Обновить клиент и документацию.
6. Пройти сценарную проверку и ручную валидацию.

## Основные риски

- Summary может терять факты, ограничения и сущности.
- Неправильная роль summary может изменить приоритет инструкций.
- Локальная оценка токенов в UI станет врать, если не перевести её на effective context.
- Потоковый и обычный режим могут разойтись, если обновление summary встроить по-разному.
- Ошибка миграции БД может затронуть существующие разговоры.

## Checkpoints для согласования

- Подтверждён `N`.
- Подтверждена трактовка порога `10 сообщений`.
- Подтверждён способ генерации summary.
- Подтверждён формат `contextCompression` в ответах.
- Подтверждено, нужен ли feature flag.

## Итоговая рекомендация

Делать минимальную и обратимую первую версию:

- отдельное summary-хранилище;
- неизменяемый хвост последних `N` сообщений;
- серверная компрессия только после успешного assistant-turn;
- одинаковая логика для JSON и SSE;
- явные метаданные компрессии для клиента.

Это даёт экономию токенов без перелома текущего API и оставляет безопасную точку для поэтапного внедрения.
